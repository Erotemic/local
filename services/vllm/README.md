# vLLM Stack Autoconfig

A self-contained starter project for standing up a local multi-GPU **vLLM + Open WebUI** stack with:

- GPU autodetection via `nvidia-smi`
- sane default configuration generation
- Docker Compose rendering
- benchmark-driven tuning for vLLM server arguments
- shared Hugging Face model cache across containers
- optional Open WebUI + Postgres services

The project is designed to be conservative by default and then become fast through measured tuning.

## What it does

`manage.py` is the main entrypoint. It can:

- inspect the current machine and record GPU inventory
- create a starter `project.yaml`
- plan GPU placement and tensor/data parallel layout for each deployment
- render `.env` and `generated/docker-compose.yml`
- bring the stack up/down through Docker Compose
- benchmark a running OpenAI-compatible endpoint
- tune a target deployment by trying multiple vLLM settings and writing the winner back into config

## Design notes

- The generated stack uses **vLLM** instead of Ollama.
- Each deployment becomes one vLLM service.
- For single-node machines, the generator prefers `--distributed-executor-backend mp`.
- Models are served through vLLM's OpenAI-compatible API.
- Open WebUI is configured through `OPENAI_API_BASE_URLS` so it can see multiple vLLM services.
- Model downloads are shared by mounting one Hugging Face cache directory into every vLLM container.

## Quick start

1. Install Python dependencies:

   ```bash
   python -m venv .venv
   source .venv/bin/activate
   pip install -e .
   ```

2. Initialize a starter config from the current machine:

   ```bash
   python manage.py init
   ```

3. Review and edit `project.yaml`.

4. Render Compose files:

   ```bash
   python manage.py render
   ```

5. Review `generated/.env`. Missing secrets are auto-generated on first render and then preserved on later renders. `HF_TOKEN` is optional and only needed for private or gated models.

6. Start the stack:

   ```bash
   python manage.py up
   ```

7. Benchmark a deployment:

   ```bash
   python manage.py benchmark --deployment default-chat
   ```

8. Tune a deployment and write the best settings back into `project.yaml`:

   ```bash
   python manage.py tune --deployment default-chat --objective balanced --apply
   ```

9. Re-render and restart:

   ```bash
   python manage.py render
   python manage.py up
   ```

## Recommended workflow

Use the safe defaults first, then tune one deployment at a time:

```bash
python manage.py init
python manage.py render
python manage.py up
python manage.py benchmark --deployment default-chat
python manage.py tune --deployment default-chat --objective balanced --apply
python manage.py render
python manage.py up
```

## Important assumptions

- `nvidia-smi` is available on the host.
- Docker Compose is available as `docker compose`.
- The NVIDIA Container Toolkit is installed and Docker is configured for GPUs.
- The vLLM image is pulled from `vllm/vllm-openai`.
- The default tuning grid is intentionally small so it does not explode into dozens of long-running trials.

## Default configuration behavior

`python manage.py init` creates a safe starter config:

- shared HF cache in `/data/service/docker/hf-cache`
- Open WebUI enabled
- Postgres enabled
- one starter deployment using an auto-selected public `Qwen3.5` model
- automatic GPU placement
- automatic TP/DP planning

If the planner can infer a parameter count from the model name, it will estimate the minimum tensor parallel size. When it cannot, it falls back to `tp=1` and asks you to tune or override manually.

## Config file structure

The main config is `project.yaml`. It contains:

- `paths`: host-side storage paths
- `runtime`: compose command and timezone
- `open_webui`: UI config
- `postgres`: database config
- `serving_defaults`: shared vLLM arguments
- `hardware`: reserved GPUs and autodetected inventory
- `deployments`: one or more served models
- `benchmark`: benchmark defaults

You usually only edit:

- `deployments[*].model`
- `deployments[*].served_model_name`
- `deployments[*].topology`
- `deployments[*].placement`
- `deployments[*].tuning`

## Common commands

Show detected GPUs:

```bash
python manage.py detect-gpus
```

Show the computed plan without rendering:

```bash
python manage.py plan
```

Render files into `generated/`:

```bash
python manage.py render
```

Start or restart the stack:

```bash
python manage.py up
```

Stop the stack:

```bash
python manage.py down
```

Benchmark a deployment's current endpoint:

```bash
python manage.py benchmark --deployment default-chat
```

Tune a deployment with an ephemeral temporary stack:

```bash
python manage.py tune --deployment default-chat --objective throughput --apply
```

## Tips

- For a single deployment on a large box, leave `gpu_count: auto`. The planner will use all available GPUs and derive DP from TP.
- For multiple deployments, set `topology.gpu_count` or `placement.gpu_indices` explicitly so they do not fight for the same GPUs.
- Keep `HF_HOME` shared across vLLM services so model downloads are reused.
- If Open WebUI has already persisted provider settings, the generated stack sets `ENABLE_PERSISTENT_CONFIG=False` so environment variables stay authoritative.

## Helper scripts

- `scripts/bootstrap.sh` sets up a local venv and installs the package.
- `scripts/example_tune.sh` shows a full tune/render/up loop.

## Caveats

- Automatic TP inference is heuristic. It assumes full-precision-style weight footprints unless you explicitly override.
- DP/TP planning across many simultaneous deployments is not a full cluster scheduler; it is a practical single-node planner.
- Tuning requires repeatedly loading the model, so large models can make `tune` runs expensive.
