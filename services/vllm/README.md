# vLLM Stack Autoconfig

A self-contained starter project for standing up a local multi-GPU **vLLM + Open WebUI** stack with:

- GPU autodetection via `nvidia-smi`
- sane default configuration generation
- Docker Compose rendering
- benchmark-driven tuning for vLLM server arguments
- shared Hugging Face model cache across containers
- optional Open WebUI + Postgres services
- autogenerated secrets on first render

## What changed in this rebuild

- `python manage.py init` works from a source checkout without needing installation first.
- `pip install -e .` works even after `generated/` exists.
- `render` generates and preserves secrets in `generated/.env`.
- the default model family is **Qwen3.5**.
- model size parsing correctly handles names like `Qwen3.5-35B-A3B`.

## Quick start

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e .

python manage.py init

# You can edit the generated project.yaml file to modify high level settings at this point.

python manage.py render
```

Review `project.yaml`, then start the stack:

```bash
python manage.py up
```

## Default behavior

`init` auto-detects your GPUs and picks a public Qwen3.5 model that should be a sane first fit for the machine:

- very small GPUs: `Qwen/Qwen3.5-0.8B`
- small GPUs: `Qwen/Qwen3.5-2B` or `Qwen/Qwen3.5-4B`
- mid-range GPUs: `Qwen/Qwen3.5-9B`
- large multi-GPU boxes: `Qwen/Qwen3.5-27B` or `Qwen/Qwen3.5-35B-A3B`

This is intentionally conservative. You can switch models later in `project.yaml` and re-render.

## Hugging Face token

You only need `HF_TOKEN` for private or gated Hugging Face models. For public models, leave it blank.

## Secrets

You do not need to hand-pick a Postgres password or vLLM API key for first boot.

`python manage.py render` creates `generated/.env` and:

- preserves existing values if already present
- generates a random `POSTGRES_PASSWORD` if missing
- generates a random `VLLM_API_KEY` if missing
- leaves `HF_TOKEN=` blank by default

## Main commands

Initialize config:

```bash
python manage.py init
```

Detect GPUs:

```bash
python manage.py detect-gpus
```

Show computed plan:

```bash
python manage.py plan
```

Render output files:

```bash
python manage.py render
```

Bring the stack up/down:

```bash
python manage.py up
python manage.py down
```

Benchmark one deployment:

```bash
python manage.py benchmark --deployment default-chat
```

Tune one deployment and write back the winner:

```bash
python manage.py tune --deployment default-chat --objective balanced --apply
```

## Notes

- The generated Open WebUI stack uses `OPENAI_API_BASE_URLS` and `OPENAI_API_KEY` so it can talk to one or more vLLM services.
- `ENABLE_PERSISTENT_CONFIG=False` is used by default so env configuration stays authoritative.
- The benchmark harness calls the OpenAI-compatible `/chat/completions` API exposed by vLLM.
- The tuning loop focuses on supported vLLM knobs that materially affect memory pressure and latency/throughput tradeoffs.


### Reconfiguration


You should be able to change your config if you need to:


```bash
cd /home/joncrall/local/services/vllm

# 1) edit the source config
$EDITOR project.yaml

# 2) regenerate the derived files
python manage.py render

# 3) stop old containers using the old config
python manage.py down

# 4) start fresh with the new config
python manage.py up
```
